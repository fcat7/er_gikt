# 数据处理架构优化：三层分离模式

## 重构时间
2026-02-27

## 重构动机

**问题诊断：**
1. ❌ **功能重复**：`adapter.py` 和 `data_process.py` 都在做列名映射和数据清洗
2. ❌ **架构不统一**：同时存在旧 Config（`config.dataset.COLUMN_MAP`）和新 TOML 配置系统
3. ❌ **职责不清晰**：Adapter 和 Builder 的边界模糊，data_process.py 承担过多逻辑

## 优化方案：选项 A - 现代化三层分离

```
┌─────────────────────────────────────────────────────────┐
│                    data_process.py                       │
│                    (流程编排主函数)                       │
│  • 加载 TOML 配置                                        │
│  • 调用 Adapter → Builder 流水线                         │
│  • 生成 metadata.json                                    │
└────────┬──────────────────────────────────────┬─────────┘
         │                                      │
         ▼                                      ▼
┌──────────────────┐                  ┌──────────────────┐
│  KTDataAdapter   │                  │  KTDataBuilder   │
│  (数据获取层)     │                  │  (模型输入层)     │
├──────────────────┤                  ├──────────────────┤
│ • load_from_toml │                  │ • build_maps     │
│ • standardize    │  标准DataFrame   │ • build_sequences│
│ • clean          │  ──────────────> │ • build_graphs   │
│ • filter         │  (uid/qid/label) │                  │
└──────────────────┘                  └──────────────────┘
  原始CSV/Parquet                      Parquet + JSON + NPY
```

## 架构对比

### 重构前

| 层级 | 职责 | 问题 |
|------|------|------|
| **data_process.py** | 流程编排 + 数据加载 + 清洗 + 特征计算 | ⚠️ 职责过重 |
| **adapter.py** | 旧 Config 的数据加载（已废弃） | ⚠️ 与新系统不兼容 |
| **builder.py** | 序列构建 + 图构建 | ✅ 职责清晰 |

**痛点：**
- `data_process.py` 包含 3 个重复函数：`load_and_standardize_data()`, `apply_filters()`, `clean_data()`
- `adapter.py` 使用旧 Config，与新 TOML 系统冲突
- 代码重复率高，维护困难

### 重构后

| 层级 | 职责 | 输入 | 输出 | 状态 |
|------|------|------|------|------|
| **KTDataAdapter** | 数据获取与标准化 | TOML配置 + 原始CSV | 标准DataFrame (uid/qid/label) | ✅ 现代化 |
| **KTDataBuilder** | 模型输入构建 | 标准DataFrame + Config | Parquet + 映射表 + 图 | ✅ 保持不变 |
| **data_process.py** | 流程编排 | 命令行参数 | 完整数据集 | ✅ 简化 |

**优势：**
- ✅ 单一职责原则（每层职责明确）
- ✅ 可测试性强（每层独立测试）
- ✅ 未来可扩展（Adapter 可支持 SQL、API 等数据源）
- ✅ 配置统一（全部使用 TOML）

## 核心改动

### 1. 现代化 `KTDataAdapter` (配置驱动)

**新接口：**
```python
class KTDataAdapter:
    @staticmethod
    def load_from_toml(toml_config):
        """
        从 TOML 配置加载数据并完成标准化处理
        
        流程：
        1. 加载原始数据 (_load_raw_data)
        2. 列名标准化 (_standardize_columns) ← Normalization Wall
        3. 应用过滤器 (_apply_filters)
        4. 数据清洗 (_clean_data)
        
        返回：标准 DataFrame (uid, qid, sid, label, timestamp, rt)
        """
```

**职责边界：**
- ✅ **负责**：CSV 加载、列名映射、过滤、清洗、排序
- ❌ **不负责**：ID 映射、序列构建、图构建（这是 Builder 的职责）

**设计模式：**
- 静态方法（无状态设计）
- 配置驱动（纯函数式）
- 单次处理（加载 → 标准化 → 输出）

---

### 2. 简化 `data_process.py` (流程编排)

**重构前：** 269 行（包含大量数据处理逻辑）

**重构后：** ~180 行（删除 90 行重复代码）

**删除的函数：**
```python
- load_and_standardize_data()  # 移到 Adapter._load_raw_data + _standardize_columns
- apply_filters()              # 移到 Adapter._apply_filters
- clean_data()                 # 移到 Adapter._clean_data
```

**新流程：**
```python
def main():
    # 1. 加载配置
    toml_config = load_dataset_config(args.dataset)
    
    # 2. 数据获取与标准化（Adapter） ← 一行代码替代原来的 3 步
    df = KTDataAdapter.load_from_toml(toml_config)
    
    # 3. 用户级采样和分割
    if args.ratio < 1.0:
        df = sampler.stratified_sample(df, ratio=args.ratio)
    train_users, test_users = splitter.split_users(df)
    
    # 4. 模型输入构建（Builder）
    builder = KTDataBuilder(target_config)
    builder.build_maps(df)
    builder.build_sequences(df_train, df_test)
    
    # 5. 生成元数据
    save_metadata(builder, output_dir)
```

**代码对比：**
```python
# 重构前（分散在 3 个函数）
df = load_and_standardize_data(dataset_config)
df = apply_filters(df, dataset_config.get('filters', {}))
df = clean_data(df)

# 重构后（一行代码）
df = KTDataAdapter.load_from_toml(dataset_config)
```

---

### 3. 保持 `KTDataBuilder` 专注于“结构构建”

Builder 职责始终清晰，聚焦在**结构与序列**，题目统计特征由 `data_process.py` 主流程基于训练集另行计算：
- `build_maps()`: 构建 ID 映射（user2idx, problem2idx, skill2idx）
- `build_sequences()`: 生成变长序列（Parquet）
- `_build_adjacency_matrices()`: Q-S, Q-Q, S-S 邻接矩阵
- `_build_domain_map()`: 技能领域聚类

## 架构优势

### ✅ **1. 单一职责原则 (SRP)**

| 组件 | 唯一职责 |
|------|---------|
| Adapter | 原始数据 → 标准 DataFrame |
| Builder | 标准 DataFrame → 模型输入 |
| data_process.py | 流程编排 |

### ✅ **2. 开闭原则 (OCP)**

**扩展示例：支持 SQL 数据源**
```python
class KTDataAdapter:
    @staticmethod
    def load_from_sql(conn_string, query, column_map):
        """未来可扩展：从数据库加载"""
        df = pd.read_sql(query, conn_string)
        df = KTDataAdapter._standardize_columns(df, column_map)
        df = KTDataAdapter._clean_data(df)
        return df
```

**无需修改：** Builder 和 data_process.py 完全不变

### ✅ **3. 依赖倒置原则 (DIP)**

```
data_process.py  ──依赖→  KTDataAdapter (抽象接口)
                          ↑
                          实现变化（CSV/SQL/API）
```

## 使用示例

### 处理 ASSIST09

```bash
python data_process.py --dataset assist09 --ratio 1.0 --stride 0
```

**流程：**
1. 加载 `config/datasets/assist09.toml`
2. **Adapter**：读取 CSV → 重命名列 → 过滤 `original=1` → 清洗
3. **Splitter**：用户级分割 (Train 80% / Test 20%)
4. **Builder**：构建 ID 映射 + 序列 + 图 + 特征
5. **输出**：`train.parquet`, `test.parquet`, `metadata.json`

### 新增数据集成本

**步骤 1：** 创建 TOML 配置
```toml
# config/datasets/new_dataset.toml
dataset_name = "new_dataset"
file_path = "path/to/data.csv"

[column_map]
uid = "student_id"
qid = "item_id"
label = "score"

[filters]
is_valid = true
```

**步骤 2：** 运行处理
```bash
python data_process.py --dataset new_dataset
```

**无需修改任何代码！** ✨
